## AI を業務システムに足すときの、よくある失敗

AI を既存の業務システムに追加しようとすると、いくつかのよくある失敗パターンがあります。

ひとつは「外側に chatbot を貼り付ける」パターンです。既存のバックエンドはそのままにして、フロントに会話 UI を追加します。見た目は新しくなりますが、chatbot は業務ロジックの中に入れないため、できることは FAQ の案内や表面的な情報提供にとどまります。ユーザーの具体的な要求に答えられないまま「担当者にお問い合わせください」で終わる、あの体験です。

もうひとつは「すべてを agent に置き換える」パターンです。既存の API を廃止して業務ロジックを LLM に委ねます。会話の自由度は上がりますが、金額計算・在庫確定・課金・認可といった正確性が要る処理まで LLM に任せると、出力が非決定的になり、責任境界が曖昧になります。テストも監査も難しく、本番環境で使い続けるには無理があります。

どちらも長続きしません。では、どう設計すればよいでしょうか。

## 自然言語フロントドアという設計原則

私が今考えている設計の中心は「AI エージェントをサービスの自然言語フロントドアとして置く」という一文に集約できます。

具体的には次の 5 つの原則として整理しています。

まず、**チームは機能（capability）を所有する**。サービスの境界は引き続きチームの責務に沿って引きます。AI を導入してもこの構造は変えません。

次に、**サービスは機能を API として公開し続ける**。REST や gRPC といった機械可読な API は廃止しません。これがシステム間連携や状態変更の正本契約であり続けます。

そして、**AI エージェントをその上に自然言語インターフェースとして追加する**。「追加」という言葉を意図的に使っています。エージェントは API の代わりではなく、API の手前に置く会話インターフェースです。ユーザーが「辛いものが食べたい」と言えば、エージェントがその意図を解釈して menu-service の catalog API を呼び出す、という役割分担です。

**system prompt はエンジニアが管理する**。役割・制約・ツール利用方針・エラー時の振る舞いを system prompt に書きます。これはソースコードと同じ扱いで、レビューとバージョン管理の対象です。

**skill は業務知識の公開面として、業務側がメンテできる形で分離する**。商品の特徴・調理時間の目安・配達エリアの注意事項といった業務用語や判断の観点は、skill ドキュメントに書きます。エンジニアでなくてもメンテナンスできる形にしておくことで、業務知識の陳腐化を防ぎます。

ここで重要なのは、**「スキルをいつ使うか」という発動条件も SKILL.md に書く**という設計です。`menu-service` では各 SKILL.md ファイルの YAML フロントマターに `activationHint` フィールドを置き、起動時に `MenuServiceConfiguration` がそれをパースして `skillActivationHints` ビーンとして提供します。`MenuApplicationService` はこのヒントを元にシステムプロンプト内のスキル発動セクションを動的に組み立てます。スキルの内容だけでなく、スキルを選ぶ判断基準自体が SKILL.md に集約されているため、業務担当者はコードを触らずにスキルの適用条件を変更できます。

そして最も重要なポイントがあります。**金額計算・在庫確定・課金・認可のような「正解が必要な処理」はコードに残す**。LLM は曖昧さの解釈や文脈の補完が得意ですが、べき等性や整合性の保証は苦手です。ここを LLM に任せると、再現性のない障害と監査不能な状態が生まれます。この分担を守ることが、エージェントを業務システムに組み込む上での最大の安全弁になります。

要するに、**agent-first ではあっても API-less ではなく、知識駆動ではあっても rule-in-prompt ではない**、というのがこの仮説の中心です。

## Arachne ── Spring になじむエージェント実行基盤

この仮説を Java と Spring Framework の現場で実現しようとすると、既製の選択肢では困ることがあります。

ツール定義をどこに書くか。会話セッションをどう管理するか。Bedrock や OpenAI の API をどう抽象化するか。Micrometer / Actuator とどう接続するか。Spring Security のスレッドコンテキストをツール実行スレッドにどう引き継ぐか。こうした問題は、Python 向けに設計されたフレームワークをそのまま持ち込んでも解決しません。

そこで、AWS が公開している Python 版 Strands Agents SDK を Java に移植し、Spring Boot 統合を追加したのが **Arachne** です。

Arachne のコアは `Agent → EventLoop → Model / Tool` というシンプルな 3 層構造になっています。`EventLoop` が LLM のレスポンスを解析してツール呼び出しを実行し、最終回答が得られるまでループします。`Model` は Bedrock の Claude / Nova 等の LLM 呼び出し抽象で、`Tool` は Java メソッドにアノテーションを付けるだけで定義できます。

Spring 統合として `ArachneAutoConfiguration` と `AgentFactory` を提供しており、Agent を Spring Bean として DI コンテナに登録できます。スキルファイルは `ResourceLoader` で読み込み、メトリクスは Micrometer に自動記録されます。既存の Spring アプリに Agent を追加しても、Controller → Application Service → Gateway → Repository の責務分離を崩す必要はありません。

セッション管理・ストリーミング・割り込み（human-in-the-loop）・構造化出力といった機能も、コアの `Agent` クラスに実装しています。

## food-delivery-demo による実証

仮説を語るだけでは不十分です。実際に動くシステムで確認する必要があります。そのために作ったのが **food-delivery-demo** です。

food-delivery-demo は、クラウドキッチンの注文・配送システムをマイクロサービスで実装したデモアプリケーションです。10 のバックエンドサービスと Next.js のカスタマー UI で構成されています。

このデモで最も重要なのは、**どのサービスにエージェントを置き、どのサービスには置かないか**という選択です。

`menu-service` はエージェントを持ちます。曖昧な注文意図（「2 人分で少し贅沢にしたい」）を具体的な商品提案に変換する役割を担います。これがまさに「自然言語フロントドア」です。エージェント呼び出しは 1 ターンで完結し、`MenuSelectionDecision`（明示的なアイテム ID・追加提案アイテム ID・適用スキルタグ・推薦理由）を返します。どのスキルを発動するかは SKILL.md の `activationHint` が決め、system prompt はその発動条件テキストを動的に受け取る構造になっています。

`delivery-service` もエージェントを持ちます。しかしこちらは少し特殊で、`registry-service` を通じて外部 ETA サービス（hermes-adapter、idaten-adapter）を動的に発見し、自社急配と比較して推奨案を出します。capability-based の動的サービス解決という仮説の重要な断面です。

`payment-service` は**エージェントを持ちません**。支払い処理は決定論的コードで実行し、LLM への依存を意図的に排除しています。課金という「正解が必要な処理」はコードで守るという原則の体現です。

こうした対比がデモ全体の設計思想を示しています。AI を前面に出しながら、正確性が要る責務はコードで守り、システム全体の責務境界は崩しません。

カスタマー UI は 3 つの公開面に分かれています。`/order` は 4 ステップの workflow UI、`/support` は会話型サポート、`/agents` は registry-backed のサービスビューワーです。なぜ 1 つの会話インターフェースに統合しないのか。それは、注文確定・サポート問い合わせ・仕様確認という 3 つの責務が、UX として異なる形を必要とするからです。
